## 도입부 — 실제로 당황했던 이야기

AI가 "이렇게 설치하세요"라고 했는데, 해보니 이미 존재하지도 않는 플러그인이었어요.

진짜 당황했던 경험이죠.... npm에도 없고 GitHub에도 없었으니까요. 처음엔 제가 명령어를 잘못 쓴 줄 알았어요. 그런데 아무리 해봐도 같은 에러가 나왔어요. 그때부터 AI가 추천한 건 무조건 검증을 하게 됐어요.

어떻게 됐냐면요... AI한테 OpenClaw에 Grok 인증 플러그인 설치 방법을 물었어요. 그러니까 "OpenClaw에서 xai-grok-auth 플러그인을 쓰는 방법이 있냐"고 물어본 거예요. 그러니까 바로 "이렇게 하시면 됩니다"라고 깔끔하게 답변을 해줬어요. npm 설치 명령어까지 있어. 너무 친절하죠. 설명도 자세하고, 예제 코드까지 붙어있고요.

그런데 그 명령어를 복사해서 터미널에 붙여넣었더니... `npm ERR! code E404` 에러가 나왔어요.

처음엔 npm 캐시 문제인 줄 알았어요. 그래서 `npm cache clean --force`로 캐시를 지우고 다시 시도해봤어요. 그런데 같은 404 에러가 나왔어요.

그다음에는 권한 문제인가 싶어서 `sudo npm install xai-grok-auth`로 해봤어요. 그런데 또 404예요. 이때 좀 이상하다는 생각이 들었어요. 왜냐면 "패키지가 없다"는 에러는 권한 문제랑은 관계가 없거든요.

그래서 그냥 npm 공식 사이트에서 직접 검색해봤어요. https://www.npmjs.com 에서 "xai-grok-auth"로 검색했어요. 그런데 결과가 없어요. 정말로 없어요.아래량량도 없고, 버전 정보도 없고요. "그냥 한 번 더 해보자" 하고 반복하다가... 결국 npm 공식 사이트에서 직접 검색해봤어요. 없어요. 정말로 없어요.

이 글에서 그 당황한 경험을 먼저 풀어드릴게요. 그리고 4가지 실제 사례와 함께, 이걸 걸러내는 5단계 검증 체크리스트도 알려드릴게요.

여기서 다룰 내용은 이 세 가지예요:

- 실제 있었던 4가지 가짜 가이드 사례
- 각각 어떻게문제 발견했는지
- 앞으로 AI 가이드 검증하는 5단계 체크리스트

## H2 #1 — 실종된 플러그인, npm에도 없던 사건

### xai-grok-auth라는 플러그인이 있대서 찾아봤는데...

어떤 AI 도구를 쓰다가 OpenClaw에 Grok 인증 기능이 필요했어요. 그래서 AI한테 물어봤어요. "OpenClaw에 Grok 인증 플러그인 설치하는 방법 알려줘요."

그러니까 바로 답변을 해줬어요. "네, xai-grok-auth라는 플러그인이 있습니다. 아래 명령어로 설치하시면 돼요."

```bash
npm install xai-grok-auth
```

완벽하게 생긴 명령어죠. 복사해서 터미널에 붙여넣었어요.

```bash
npm install xai-grok-auth
# npm ERR! code E404
# npm ERR! 404 Not found: xai-grok-auth
```

에러 메시지가 나오면서 "그냥 한 번 더 해보자" 하고 반복했어요. npm 캐시를 지우고 해보고, 권한 문제인가 싶어서 sudo로 해보고... 그런데 다 같은 404 에러였어요.

결국 npm 공식 사이트에서 직접 검색해봤어요. npmjs.com에서 "xai-grok-auth"로 검색했어요. 결과가 없어요. 정말로 없어요.

### 왜 이렇게 된 걸까

이건 AI가 "있는 것 같지만 없는 것"을 만들어낸 거예요. Hallucination, 일종의 환각 현상이에요. 학습 데이터에 이 플러그인에 대한 정보가 없는데, AI가 그럴듯하게 만들어낸 거예요.

"아, 이 플러그인이 있구나" 하는 느낌으로 답변을 해요. 근데 실제로는 없어요.

> 💡 여기서 핵심: AI가 "존재한다"고 했더라도, npm이나 GitHub에서 먼저 확인하세요.

## H2 #2 — GitHub 404, 복사했는데 레포가 없었어

### mcparkMCP/openclaw-xai-auth, clone하려는데 fatal error

한 번 당했으니 이번엔 좀 더신중하게 해보자 했어요. 그런데 또 있었어요.

OpenClaw 관련해서 Instagram 자동화 관련 문서를 정리하려고 AI한테 물어봤어요. 그러니까 특정 GitHub 레포를 추천했어요. "mcparkMCP/openclaw-xai-auth"라는 레포가 있다고 했어요. 설명까지 붙어서 정말 있는 것처럼 보여요.

```bash
git clone https://github.com/mcparkMCP/openclaw-xai-auth.git
```

복사해서 터미널에 붙여넣었어요. 그런데...

```
fatal: repository 'https://github.com/mcparkMCP/openclaw-xai-auth' does not exist
```

레포가 없어요. 404 Not Found예요.

### 어떻게 발견했나

그냥 clone 안 되는 걸 보고 알았어요. 그런데 더 빠른 방법을 나중에 알았어요.

```bash
# GitHub 레포 존재 확인 — clone 전에 먼저 체크
git ls-remote https://github.com/mcparkMCP/openclaw-xai-auth 2>&1
# fatal: repository does not exist

# curl로 HTTP 상태 확인
curl -s -o /dev/null -w "%{http_code}" https://github.com/mcparkMCP/openclaw-xai-auth
# 404
```

이 방법을 알면 시간 낭비 없어요. clone하기 전에 확인하면 돼요.

### 왜 위험한가

이게 위험한 건, 설명까지 완벽하게 붙어서 오는 거예요. URL도 생기고, 설명도 있어요. 그냥 믿게 되거든요. 그런데 실제로는 없는 거예요.

존재하는 것처럼 보이는 것만으로 신뢰하는 거예요. 이게 AI의 문제예요. 그럴듯한 정보를 만들어내거든요.

| 확인 방법 | 명령어 | 속도 |
|---|---|---|
| git ls-remote | `git ls-remote <url>` | ⚡ 1초 |
| curl HTTP 상태 | `curl -I <url>` | ⚡ 1초 |
| 브라우저에서 수동 확인 | 직접 접속 | 🐢 30초 |

## H2 #3 — CLI 옵션이 없어, --auth-choice xai-oauth의 순간

### AI가 '--auth-choice xai-oauth' 옵션을 사용하라 했는데...

이건 진짜 어이없었어요. 다른 경험이랑은 조금 달라요.

OpenClaw 문서를 정리하려고 AI한테 물어봤어요. 특정 인증 옵션 설정 방법을 물어본 거예요. 그러니까 이렇게 답변을 해줬어요.

"OpenClaw에서 `--auth-choice xai-oauth` 옵션을 사용하세요. 이렇게 하면 됩니다..."

그리고 실제 명령어 예시까지 보여줬어요. 너무 확신이충분해서, 당연히 있는 옵션이라고 생각했어요.

그런데 `openclaw --help`로 확인해봤어요. 그러니까 `openclaw --help`의 전체 출력을 grep으로 뒤져봤어요. `-i` 옵션으로 대소문자 구분 없이 검색했어요. 그런데 어디에도 "xai-oauth"라는 문자열이 없었어요.

솔직히 처음엔 도구 자체가 잘못된 건가 싶었어요. openclaw가 최신 버전이 아닌가 싶어서 다시 설치해봤어요. 그런데 그래도 같은 결과였어요. 어느 세션에서 해봐도, 어느 옵션으로 해봐도, xai-oauth는 없었어요. 이때 가서야 알았어요. AI가 만들어낸 옵션이구나 하고요. 사실 이 상황 자체가 좀 웃기기도 해요. "이 옵션 있어"라고 하면서, 실제로는 없는 옵션을 추천하는 거니까요.

솔직히 처음엔 도구 자체가 잘못된 건가 싶었어요. openclaw가 최신 버전이 아닌가 싶어서 다시 설치해봤어요. 그런데 그래도 같은 결과였어요. 어느 세션에서 해봐도, 어느 옵션으로 해봐도, xai-oauth는 없었어요.

```bash
# AI가 권장한 옵션이 실제 있는지 확인
openclaw --help 2>&1 | grep -i "xai-oauth"
# (아무 출력 없음)
```

없어요. 정말로 없어요. 전체 help 출력에서 grep으로 찾아봐도 없었어요.

### 어떻게 발견했나

솔직히 말하면... 이거 발견했을 때 진짜 어이없었어요. AI가 확신 있게 말하니까 당연히 있는 줄 알았거든요. 그런데 실제로 해보니까 없었어요.

```bash
# 전체 도움말 확인
openclaw --help
# 이 출력에서 xai-oauth 검색 — 결과 없음
```

이 мом엔트에서 깨달았어요. AI는 "있는 것처럼" 말할 뿐, 실제로 있는지는 모른다는 거예요.

### 이 교훈

CLI 도구 문서는 항상 `--help`로 직접 확인하세요. AI가 뭐라고 하든, 내가 직접 확인하는 게 빨라요.

## H2 #4 — 가짜 디렉토리 사이트, URL 자체가 존재하지 않았어

### AI가 '공식 디렉토리'라고 했는데 접속이 안 되더라구요

이건 좀 더 교활한 경우였어요.

AI한테 OpenClaw 플러그인을 어디서 찾아야 하는지 물어봤어요. 그러니까 "공식 플러그인 디렉토리가 있어요" 라고 답변을 해줬어요. 그리고 URL까지 보내줬어요.

예를 들면 이런 거예요:

```
https://openclawdir.com/plugins/xai-grok-auth
```

보기에는 그럴듯해요. "공식 디렉토리"라는 말도 붙고, URL 구조도 그럴듯하고요. 그런데 접속해볼 엄두가 안 났어요. 이번엔 먼저 확인해보자 해서요.

```bash
# curl -I로 HTTP 헤더만 확인
curl -I https://openclawdir.com/plugins/xai-oauth 2>&1 | head -1
# HTTP/2 404
```

404예요. 해당 URL이 없어요. 그런데 더 심한 건, 그 도메인 자체가 존재하지 않는 경우도 있어요.

```bash
# 도메인 자체가 없는 경우
curl -I https://openclawdir.com 2>&1
# Could not resolve host: openclawdir.com
```

이건 그냥 AI가 만들어낸 도메인인 거예요. URL을 복사하기 전에 반드시 Hover로 실제 도메인 확인하세요.

### 방지법

URL을 받으면 바로 클릭하지 말고, 먼저 도메인만 확인하세요. ".com"이나 ".io"가 실제 서비스인지 확인하는 습관이 필요해요.

## H2 #5 — 왜 AI는 거짓말을 할까

### 이건 버그가 아니라 특성입니다

여기까지 읽으시면 의문이 드는 게 있을 거예요. "그럼 AI는 왜 이렇게 하는 거죠?"

이건 버그가 아니에요. 특성이에요. Hallucination, 일종의 환각 현상이라고 해요.

AI는 학습 데이터에 없는 내용을 그럴듯하게 생성해내요. 모르는 걸 알 것 같은 톤으로 말하는 거예요. 확신이 넘치거든요. 근데 사실이 아닌 경우도 있어요.

### 왜 그럴까

학습 데이터 바닥이 한 가지예요. 특정 패키지나 버전에 대한 정보가 학습 데이터에 없으면, AI는 그럴듯한 정보를 만들어내요. 마치 옛날 백과사전 편집자가 모르는 걸 수도 있다고 생각하면서도 그냥 쓴 것과 비슷해요.

자신감 과잉도 있어요. AI는 "모르겠습니다"라고 말하는 것보다, 모르는 걸 그럴듯하게 대답하는 게 기본 설정이에요. 그래서 더 그럴듯해 보여요.

### 비유로 이해하기

이걸 이해하려면 비유를 하나 들어볼게요.

AI는 **자신감이 넘치는 하지만 사실 잘 모르는 동료**와 같아요. 어떤작업을 맡겨도 "네, 알겠니다"라고 해요. 그런데 실제로 해보면 엉망인 경우가 있어요. "아, 그건 제가 잘못 알아들었네요"라고 하는 게 아니라, 계속 확신 있게 말하는 거예요.

이 동료와 일하려면 어떻게 해야 할까요? 모든 걸 직접 검증해야 하잖아요. AI도 마찬가지예요.

어떤 모델은 "xai-grok-auth"라 하고, 어떤 모델은 "grok-xai-auth"라 해요. 둘 다 없지만요. 이러면 오히려 더 헷갈려요. 어떤 건 세 개를 조합하고, 어떤 건 두 개를 조합하고... 하나같이 다 없어요.

**Q. AI가 추천하는 패키지는 전부 의심해야 하나요?**

전부는 아니에요. npm trend, GitHub star, 마지막 commit 날짜 등을 확인하면 도움이 돼요. 아무도 쓰지 않는 패키지면 의심해보고, 유명한 패키지면 믿어도 괜찮아요.

**Q. 검증 귀찮은데 간편한 방법 없나요?**

있다! 아래 5단계 체크리스트가 있어요. 간단한 명령어 몇 개로 대부분 걸러낼 수 있어요.

## 5단계 검증 체크리스트

### "그냥 npm view랑 git ls-remote만 하면 돼요"

이 순서대로 검증하면 대부분의 가짜 가이드를 걸러낼 수 있어요.

사실 처음엔 검증 체크리스트 같은 거 없이 그냥 됐어요. npm install이 되면 "있고"이고, 안 되면 "없는 건가" 하고 생각했어요. 그런데 문제가 하나 있었어요. 가끔 npm install이 되는 경우가 있어요. 근데 그건 homoglyph 공격으로 다른 패키지를 설치하는 거였어요. 이름이 비슷하게 생긴 악성 패키지가 가끔 있거든요.

그래서 단순히 install이 되는지만으로 믿으면 안 돼요. npm view로 정확한 정보를 확인해야 해요.

**1단계: npm registry 확인**

```bash
# npm에 실제 있는지 확인
npm view xai-grok-auth

# 404면 없음
npm view xai-grok-auth 2>/dev/null && echo "EXISTS" || echo "NOT FOUND"
```

**2단계: GitHub 레포 확인**

```bash
# git ls-remote로 확인

```

```bash
# AI가 권장한 옵션이 실제 있는지 확인
openclaw --help 2>&1 | grep -i "xai-oauth"
# 아무 출력 없으면 없음
```

**4단계: 공식 문서 원문 확인**

AI 추천 대신 공식 문서를 직접 방문하세요. URL을 받았다면 브라우저에서 직접 접속해서 확인하세요.

**5단계: 교차 검증**

다른 AI나 검색엔진에서 같은 키워드로 검색해서 비교하세요. 여러 곳에서 같은 답이 나오면 신뢰도가 높아져요.

### 검증 체크리스트 순서도

```
AI가 플러그인/URL/옵션을 추천함
    ↓
1단계: npm view로 패키지 확인
    ↓ (없으면) → 가짜, 종료
    ↓ (있으면)
2단계: git ls-remote로 레포 확인
    ↓ (없으면) → 가짜, 종료
    ↓ (있으면)
3단계: CLI --help로 옵션 확인
    ↓ (없으면) → 가짜, 종료
    ↓ (있으면)
4단계: 공식 문서 직접 확인
    ↓ (다르면) → 가짜, 종료
    ↓ (같으면)
5단계: 교차 검증
    ↓ (일치) → ✅ 믿어도 됨
```

| 검증 방법 | 명령어 | 속도 | 신뢰도 |
|---|---|---|---|
| npm registry | `npm view <pkg>` | ⚡ 빠름 | ★★★★★ |
| GitHub 레포 | `git ls-remote <url>` | ⚡ 빠름 | ★★★★★ |
| CLI --help | `<cmd> --help \| grep` | ⚡ 빠름 | ★★★★★ |
| 공식 문서 | 브라우저 직접 접속 | 🐢 보통 | ★★★★★ |
| 교차 검증 | 다른 AI/검색엔진 | 🐢 보통 | ★★★ |

## 마무리 — 정리하면

여기까지가 4가지 실제 사례와 5단계 검증 체크리스트였어요.

AI는 좋은 도구예요. 그런데 "존재한다"고 했더라도 실제로 존재하는지는 내가 직접 확인해야 해요.

솔직히 말하면... 솔직히 이거 발견하고 나서 한 동안은 AI를 안 믿을 지경이었어요. 그런데 그건 좀 극단적이었어요. AI는 많은 상황에서 진짜 유용하거든요. 문제는 "검증 없이 바로 신뢰하는 것"이에요. 이 습관만 버리면, AI는 여전히 좋은 도구예요.

핵심은 "AI가 뭐라고 하든, 내가 직접 확인하는 것". 이것만 기억하면 돼요. 이거 모르고 몇 번 당하고 나니, 이제는 무조건 검증하게 됐어요.

사실 저도 처음엔 AI를 믿고 말았거든요... 그런데 몇 번 당하고 나니 검증 습관 생겼어요. 지금은 npm install이나 git clone하기 전에, 먼저 확인하는 게 기본이 됐어요.

실제로 속당한 경험 있으면 댓글로 공유해요. 같이 웃어요 ㅎㅎ

저도 몇 번이나 당하고 나서야 이 패턴을 알게 됐어요. 사실 이 글이나마 쓸 상황이 됐다는 게... 그때 그 당황이 아직도 생생해요. npm ERR! 404가 나왔을 때, "이게 뭔 상황이야" 한 게 이 글의 시작이었으니까요.

그래서 이 글이 도움이 되셨다면, 같이 검증 습관 들여요. AI는 도구일 뿐이에요. 최종 검증을 하는 건 언제나 우리 자신이에요.
