## 도입부

디스코드에 AI 봇 3개를 넣으면 자동으로 협업이 될까요?

저도 처음엔 약간 그렇게 기대했습니다. 오픈클로가 있고, 그록이 있고, Claude7이 있으면... 그냥 서로 알아서 멘션하고, 조사하고, 글 쓰고, 검수하고, 최종 보고까지 해줄 것 같았거든요.

그런데 실제로 해보니까 아니었습니다.

사실 이 부분이 제일 현실적이었습니다... 봇이 많아지면 똑똑해지는 게 아니라, 먼저 복잡해집니다.

봇을 3개 넣는 것과, 봇 3개가 **같은 작업 흐름 안에서 서로 일하는 것**은 완전히 다른 문제였어요. 겉으로 보면 단순합니다. 오픈클로가 그록에게 리서치를 시키고, 그록이 결과를 내고, Claude7이 검수하고, 다시 오픈클로가 정리하면 끝. 말로는 너무 쉽죠.

근데 막상 Discord 채널에서 돌려보면 이런 문제가 바로 나옵니다.

- `@오픈클로`라고 타이핑했는데 실제 멘션이 아니어서 봇이 못 알아듣습니다.
- 오픈클로가 그록을 부르라고 했는데, 그록이 엉뚱하게 이미지 생성으로 빠집니다.
- Grok 리서치가 너무 오래 걸려서 timeout이 납니다.
- Claude7은 limit 때문에 응답이 늦거나 끊깁니다.
- 어떤 봇이 어떤 단계까지 했는지 사람이 다시 추적해야 합니다.

사실 여기서 제일 크게 느낀 건 이거였습니다.

**멀티 에이전트 자동화는 모델 문제가 아니라 운영 문제다.**

GPT가 좋냐, Grok이 좋냐, Claude가 좋냐도 중요합니다. 그런데 그보다 먼저 잡아야 하는 게 있어요. 누가 시작하고, 누가 넘겨받고, 실패하면 누가 대신하고, 최종 결과는 어디에 저장되는지. 이게 없으면 아무리 좋은 모델을 붙여도 그냥 채팅방이 복잡해질 뿐입니다.

이번 글은 제가 Discord에서 오픈클로, 그록, Claude7을 실제로 연결하면서 겪은 과정입니다. 성공한 부분도 있고, 실패한 부분도 있습니다. 특히 실패가 꽤 중요했어요. 왜냐하면 자동화는 성공할 때보다 실패할 때 구조가 드러나거든요.

## 처음 목표는 단순했습니다

제가 만들고 싶었던 구조는 이거였습니다.

```text
대표님
  ↓
OpenClaw
  ↓
Grok
  ↓
Claude7
  ↓
OpenClaw
  ↓
최종 보고
```

조금 더 풀면 이런 역할입니다.

| 봇       | 역할                                              |
| -------- | ------------------------------------------------- |
| OpenClaw | 오케스트레이터, 작업 분배, 저장, 최종 보고        |
| Grok     | 리서치, Blog Research Brief, 이미지/영상 프롬프트 |
| Claude7  | 검수, 논리 점검, 빠진 부분 확인                   |

겉으로 보면 팀 같습니다. 오픈클로는 PM이고, 그록은 리서처이고, Claude7은 리뷰어인 셈이죠.

그런데 사실 여기서 바로 첫 번째 착각이 생깁니다. 봇 이름을 사람이 이해하는 것처럼 봇도 이해할 거라고 생각하는 거예요. 예를 들어 제가 Discord에서 이렇게 썼습니다.

```text
@오픈클로 그록에게 리서치 시켜줘
```

눈으로 보면 멘션처럼 보입니다. 그런데 Discord API 기준으로는 그냥 텍스트일 수 있습니다. 실제 멘션 객체가 잡히지 않으면 봇 입장에서는 아무도 부르지 않은 겁니다.

이게 처음엔 진짜 헷갈립니다. 화면에서는 파란색으로 보이는 것 같고, 사람은 “멘션했는데?”라고 느끼는데, 로그를 보면 `mentions: []`로 나와요. 그러면 봇은 조용합니다.

처음엔 이걸 보고도 믿기지가 않았습니다... 분명히 부른 것 같은데, 시스템 입장에서는 아무도 부르지 않은 거니까요.

그래서 첫 번째 규칙이 생겼습니다.

**봇을 부를 때는 이름을 타이핑하지 말고, Discord 자동완성에서 봇을 선택해야 한다.**

아주 사소해 보이죠? 그런데 이거 하나 때문에 자동화가 아예 시작도 안 됩니다.

## 실제 멘션과 텍스트 멘션은 다릅니다

Discord에서 봇 연동을 할 때 가장 먼저 확인해야 할 것은 메시지 안의 `mentions` 배열입니다.

실제 멘션이면 대략 이런 식으로 잡힙니다.

```json
{
  "content": "<@1506668791147532398> 작업 시작해",
  "mentions": [
    {
      "id": "1506668791147532398",
      "username": "Blue openclaw Testing"
    }
  ]
}
```

반대로 그냥 타이핑하면 이렇게 됩니다.

```json
{
  "content": "@오픈클로 작업 시작해",
  "mentions": []
}
```

사람 눈에는 둘 다 비슷합니다. 그런데 봇 입장에서는 완전히 다릅니다.

저는 이걸 확인하고 나서야 왜 어떤 채널에서는 응답이 있고, 어떤 채널에서는 응답이 없는지 이해했습니다. 블로그 채널에서 `@오픈클로`라고 쳤는데 오픈클로가 계속 “입력 중”만 보이고 결과가 안 나왔던 것도, 실제로는 새 작업이 제대로 안 잡힌 흐름이 섞여 있었습니다.

그러니까 여기서부터는 감으로 보면 안 됩니다... 로그가 답입니다.

그래서 멀티 봇 구조에서는 “멘션이 됐는가?”를 감으로 보면 안 됩니다. 로그로 봐야 합니다.

```bash
openclaw message read --channel discord --account default \
  --target channel:일반채널ID --limit 5 --json
```

여기서 `mentions`가 비어 있으면 그건 멘션이 아닙니다. 이건 진짜 기본인데, 실제 운영에서는 제일 먼저 걸립니다.

## 두 번째 문제: 봇끼리 서로를 무시합니다

그 다음 문제는 봇 메시지 필터였습니다.

많은 Discord 봇은 기본적으로 다른 봇의 메시지를 무시합니다. 이유는 이해됩니다. 봇끼리 서로 답하다가 무한 루프가 생길 수 있거든요.

그런데 멀티 에이전트 협업에서는 이 기본값이 오히려 문제입니다. 오픈클로가 그록에게 지시하고, 그록이 Claude7에게 넘기고, Claude7이 다시 오픈클로에게 보고해야 하는데, 봇 메시지를 전부 무시하면 체인이 끊깁니다.

그래서 저는 허용할 봇 ID를 명시했습니다.

```text
OpenClaw bot ID  = 1506668791147532398
Grok bot ID      = 1507038641883582545
Claude7 bot ID   = 1506686495174557736
```

핵심은 “모든 봇 메시지를 허용”이 아닙니다. 그건 위험합니다. 필요한 봇만 허용해야 합니다.

```text
허용:
- OpenClaw
- Grok
- Claude7

차단:
- 그 외 봇
- 자기 자신
- 멘션 없는 봇 메시지
```

이렇게 해야 봇끼리 협업은 가능하면서도 무한 루프는 줄일 수 있습니다.

사실 이 지점이 중요합니다. 멀티 에이전트는 “열어두기”가 아니라 “좁게 허용하기”예요. 자동화는 넓게 열면 금방 사고 납니다.

## 세 번째 문제: 한 메시지에 봇을 너무 많이 멘션하면 꼬입니다

처음에는 이런 식으로 지시하고 싶었습니다.

```text
@오픈클로
Grok에게 리서치 시키고,
Claude7에게 검수 시키고,
끝나면 다시 OpenClaw에게 보고해.
```

사람이 보기에는 완벽한 지시입니다. 그런데 실제 Discord에서는 이 메시지 안에 Grok과 Claude7의 실제 멘션이 들어가면, 오픈클로만 읽는 게 아니라 Grok과 Claude7도 동시에 반응할 수 있습니다.

그러면 어떤 일이 생기냐면요.

- 오픈클로가 아직 작업 계획을 세우기도 전에 Grok이 먼저 반응합니다.
- Grok이 리서치 요청이 아니라 다른 문맥으로 해석합니다.
- Claude7이 중간 단계인데도 먼저 검수를 시작합니다.
- 채널에는 여러 봇의 “처리 중” 메시지가 동시에 뜹니다.

이거 직접 보면 꽤 정신없습니다. 자동화가 된 것 같지만 사실은 각자 따로 움직이는 겁니다.

그래서 규칙을 바꿨습니다.

**대표님이 처음 부를 때는 OpenClaw만 멘션한다.**

그 다음부터는 OpenClaw가 필요한 봇을 하나씩 호출합니다.

```text
대표님 → OpenClaw만 멘션
OpenClaw → Grok 멘션
Grok → 결과 또는 완료 보고
OpenClaw → Claude7 멘션
Claude7 → 검수 결과 보고
OpenClaw → 최종 저장/보고
```

이렇게 하면 채널은 조금 느려 보여도 훨씬 안정적입니다. 멀티 봇 자동화에서 “한 번에 다 부르기”는 생각보다 위험합니다.

## 네 번째 문제: Grok이 이미지로 오해했습니다

이번 테스트에서 제일 재밌었던 오류가 하나 있었습니다.

제가 Grok에게 Blog Research Brief를 요청하면서 형식 안에 이런 항목을 넣었습니다.

```text
6. 이미지/릴스 소재 아이디어
```

의도는 간단했습니다. “블로그를 쓰고 나서 어떤 이미지나 릴스 소재로 재활용할지 아이디어를 글로 정리해줘”라는 뜻이었어요.

그런데 Grok 봇 라우터는 `이미지`, `릴스`라는 단어를 보고 media route로 분류했습니다. 결과적으로 텍스트 리서치가 아니라 이미지 생성으로 빠졌습니다.

이게 자동화에서 정말 무서운 지점입니다. 사람은 문맥을 봅니다. “이미지 소재 아이디어”는 당연히 글로 답하라는 뜻이죠. 하지만 단순 키워드 라우터는 `이미지`라는 단어 하나만 보고 “이미지 생성 요청이네?”라고 판단할 수 있습니다.

그래서 라우팅 규칙을 바꿨습니다.

```text
기존:
- "이미지" 또는 "릴스"가 있으면 media route

수정:
- "이미지/릴스" 키워드가 있어도
- "생성", "만들어줘", "그려줘", "업로드" 같은 action 키워드가 있을 때만 media route
- "리서치", "brief", "근거", "출처"가 있으면 research route 우선
```

이렇게 바꾸니까 “이미지 소재 아이디어”는 리서치로 가고, “이미지 만들어줘”는 media로 갑니다.

사실 이건 작은 수정처럼 보이지만, 자동화에서는 꽤 큽니다. 자동화의 대부분 오류는 거창한 알고리즘 문제가 아니라 이런 **단어 하나의 오해**에서 시작하거든요.

## 다섯 번째 문제: timeout은 실패가 아니라 fallback 신호여야 합니다

Grok 리서치에서 또 하나 문제가 나왔습니다.

처음에는 `max_turns exceeded`가 났고, 그다음에는 `grok cli timed out after 120s`가 났습니다. 그러면 자동화가 멈춥니다. 채널에는 “실패”라고 뜨고, 다음 단계로 진행하지 않습니다.

그런데 블로그 자동화 관점에서 보면 이건 좋지 않습니다.

왜냐하면 대표님이 자러 간 상태라면, 여기서 멈추면 끝이거든요. 아침에 일어나서 보면 “Grok 실패” 하나만 남아 있고, 블로그 3편은 아무것도 안 올라가 있는 상태가 됩니다.

자동화라면 이렇게 움직여야 합니다.

```text
Grok 리서치 요청
  ↓
정상 응답이면 그 결과 저장
  ↓
timeout이면 GPT/OpenClaw fallback brief 생성
  ↓
02-grok-research-brief.md 저장
  ↓
03-draft.md 작성으로 계속 진행
```

핵심은 이겁니다.

**fallback이 저장되면 성공으로 간주하고 다음 단계로 가야 한다.**

fallback은 완벽한 리서치가 아닐 수 있습니다. 하지만 자동화 전체를 멈추는 것보다는 낫습니다. 특히 블로그 자동 발행처럼 “오늘 3편”이라는 목표가 있는 작업에서는 더 그렇습니다.

물론 아무 내용이나 쓰면 안 됩니다. fallback에는 반드시 표시가 있어야 합니다.

```text
Grok timeout으로 인해 OpenClaw fallback brief로 진행
확인 필요 항목:
- 외부 출처 검증
- 최신 API 변경 여부
- 실제 로그 기반 여부
```

이렇게 하면 나중에 사람이 검수할 때도 어디까지가 실제 리서치이고, 어디부터가 fallback인지 구분할 수 있습니다.

## 역할 분리가 진짜 핵심입니다

이번 구조를 정리하면 이렇게 됩니다.

| 단계              | 담당             | 실패 시                               |
| ----------------- | ---------------- | ------------------------------------- |
| 요청 접수         | OpenClaw         | 사람에게 질문하지 말고 작업 폴더 생성 |
| 리서치            | Grok             | timeout 시 GPT/OpenClaw fallback      |
| 초안 작성         | OpenClaw/GPT     | 글자 수 부족 시 확장                  |
| 검수              | Claude7          | limit이면 OpenClaw gpt-5.5 검수       |
| GitHub Pages 발행 | OpenClaw         | git 상태/빌드 로그 보고               |
| Naver 업로드      | local Playwright | 실패 스크린샷 저장                    |
| Instagram Reels   | Graph API        | container status 로그 저장            |

여기서 중요한 건 모든 단계에 “대체 담당자”가 있다는 점입니다.

Grok이 안 되면 멈추는 구조는 자동화가 아닙니다. Claude7 limit이 걸리면 멈추는 구조도 자동화가 아닙니다. 네이버 업로드가 실패하면 스크린샷 없이 “실패했습니다”만 남기는 것도 자동화라고 보기 어렵습니다.

자동화는 성공 루트보다 실패 루트가 더 중요합니다.

저는 이번에 이걸 꽤 강하게 느꼈습니다. 성공할 때는 뭐든 좋아 보입니다. 그런데 timeout, 멘션 실패, route 오분류가 한 번씩 나오면 그제서야 진짜 구조가 보입니다.

## 채널 설계도 필요합니다

처음엔 모든 걸 한 채널에서 하고 싶었습니다. 일반 채널에서 오픈클로 부르고, 그록 부르고, Claude7 부르면 되겠지 싶었어요.

그런데 실제 운영에서는 채널을 나누는 게 좋습니다.

| 채널                | 역할                    |
| ------------------- | ----------------------- |
| 일반                | 테스트, 짧은 지시       |
| 블로그              | 블로그 하네스 실행      |
| 자동화              | 배치 작업, 발행 상태    |
| 오픈클로-설정       | 라우팅/모델/권한 설정   |
| grok-media-research | Grok 리서치/이미지 전용 |

특히 `grok-media-research` 채널은 멘션 없이도 Grok이 바로 알아듣는 구조로 둘 수 있습니다. 반대로 일반 채널에서는 멘션 없는 대화는 OpenClaw만 받게 하고, Grok과 Claude7은 실제 멘션이 있을 때만 움직이게 하는 편이 안전합니다.

이렇게 하지 않으면, 그냥 대화하다가 갑자기 Grok이 반응하거나 Claude7이 끼어들 수 있습니다. 처음엔 신기하지만, 자동화가 길어질수록 이건 피로해집니다.

## 실제로 필요한 체크리스트

Discord 3개 봇 연동을 하려면 최소한 이것부터 봐야 합니다.

```text
1. 봇 ID를 모두 기록했는가?
2. 실제 멘션이 mentions 배열에 잡히는가?
3. 봇 메시지 허용 목록이 있는가?
4. 자기 자신 메시지는 무시하는가?
5. 멘션 없는 봇 메시지는 무시하는가?
6. route=research/media/light 분기가 안정적인가?
7. Grok timeout fallback이 있는가?
8. Claude7 limit fallback이 있는가?
9. 각 단계별 파일 저장이 있는가?
10. 최종 보고에 파일 존재 확인이 포함되는가?
```

이 중 하나라도 빠지면 자동화가 중간에 멈춥니다.

저는 특히 7번과 8번이 중요하다고 봅니다. 모델은 언제든 느려질 수 있고, limit이 걸릴 수 있고, API가 이상하게 응답할 수 있습니다. 그때 사람을 깨우는 구조가 아니라, fallback으로 계속 진행하는 구조가 필요합니다.

물론 계정에 직접 영향을 주는 네이버 발행, 인스타 릴스 업로드 같은 작업은 조심해야 합니다. 일일 한도도 지켜야 하고, 실패 로그도 남겨야 합니다. 자동화는 무조건 많이 하는 게 아니라, **정해진 한도 안에서 끝까지 책임지는 것**에 가깝습니다.

## 블로그 자동화까지 연결하면 이렇게 됩니다

제가 최종적으로 원하는 구조는 이겁니다.

```text
대표님: "오늘 블로그 3편 발행해"
  ↓
OpenClaw: 후보 글 선택 / 없으면 생성
  ↓
Grok: 리서치 brief / 실패 시 fallback
  ↓
OpenClaw: 9,000~11,000자 초안 작성
  ↓
Claude7: 검수 / limit이면 gpt-5.5 검수
  ↓
OpenClaw: GitHub Pages push
  ↓
OpenClaw: Naver 3편 업로드
  ↓
OpenClaw: Instagram Reels 3개 업로드
  ↓
최종 보고
```

여기서 대표님이 중간에 “다음”, “계속”, “저장해”를 누르면 안 됩니다. 그건 자동화가 아니라 반자동입니다.

자동화라면 중간 결과를 저장하고, 실패하면 대체 루트로 가고, 최종적으로 무엇이 됐고 무엇이 막혔는지를 보고해야 합니다.

그래서 저는 이번 테스트의 결론을 이렇게 잡았습니다.

**AI 봇 3개를 붙이는 것보다, 실패해도 다음 단계로 가는 하네스를 만드는 게 더 중요하다.**

## 마무리

디스코드 3개 봇 연동은 생각보다 재밌습니다. 오픈클로가 그록에게 일을 시키고, 그록이 Claude7을 부르고, Claude7이 다시 오픈클로에게 보고하는 걸 채널에서 직접 보면... 약간 미래의 작업실을 보는 느낌이 납니다.

그런데 동시에 현실적입니다. 멘션 하나 잘못되면 안 움직이고, 키워드 하나 잘못 잡으면 이미지 생성으로 빠지고, 모델이 120초 안에 답을 못 하면 전체 자동화가 멈춥니다.

그래서 결국 중요한 건 멋진 데모가 아닙니다.

**실패해도 멈추지 않는 구조.**

이게 진짜입니다.

저는 앞으로 블로그 자동화도 이 방향으로 잡으려고 합니다. Grok이 안 되면 OpenClaw가 대신 brief를 만들고, Claude7 limit이 걸리면 gpt-5.5가 검수하고, 네이버가 막히면 스크린샷과 로그를 남기고, 인스타가 실패하면 container status를 저장하는 식으로요.

처음엔 봇 3개를 붙이는 게 목표였는데, 직접 해보니까 목표가 바뀌었습니다.

봇을 많이 붙이는 게 아니라, **대표님이 자러 가도 아침에 결과가 남아 있는 구조**를 만드는 것.

그게 진짜 자동화였습니다.

한 번 해보시면 압니다. 봇이 대답하는 순간보다, 봇이 실패했는데도 다음 단계로 넘어가는 순간이 더 든든합니다.
