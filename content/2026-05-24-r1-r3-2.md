## 최종 자동 재인증은 로그인 버튼 하나의 문제가 아니다

### 인증이 됐다는 말보다 실제 작업이 되는지가 중요하다

R1 R3 최종 자동 재인증이라는 말은 꽤 기술적으로 들립니다. 그런데 핵심은 단순합니다. “정말 다시 인증됐는가?”가 아니라 “인증된 상태로 실제 작업이 끝까지 이어지는가?”를 확인하는 일입니다. 자동화 하네스에서는 이 차이가 큽니다.

로그인 화면을 통과했다고 해서 모든 것이 끝나는 것은 아닙니다. 세션이 살아 있어도 특정 도구 호출은 실패할 수 있고, 파일은 읽히지만 쓰기는 막힐 수 있고, R1 경로는 정상인데 R3 경로는 오래된 권한을 보고 있을 수 있습니다. 그래서 재인증은 단순 로그인 확인이 아니라 실제 작업 흐름 확인이어야 합니다.

사실 인증 문제는 조용히 숨어 있다가 마지막에 터지는 경우가 많습니다. 초안은 만들어진 것처럼 보이는데 저장이 안 됐거나, 검수 완료라고 했는데 실제 본문을 읽지 못했거나, 발행 직전에서 권한 오류가 나는 식입니다. 이럴 때 사용자는 “왜 끝났다고 했는데 결과물이 없지?”라고 느낍니다.

솔직히 자동화에서 가장 피곤한 실패가 바로 이것입니다. 오류가 크게 보이면 바로 고치면 됩니다. 그런데 완료처럼 보였는데 실제 산출물이 없으면 원인을 다시 거슬러 올라가야 합니다. 그래서 R1 R3 최종 자동 재인증은 겉으로 보이는 인증 상태보다 작업 증거를 확인해야 합니다.

### R1과 R3는 같은 상태를 본다고 가정하면 안 된다

R1과 R3를 나눠 부르는 이유는 두 경로가 서로 다른 역할이나 상태를 가질 수 있기 때문입니다. R1은 초안 생성, 기본 실행, 첫 산출물 생성에 가까울 수 있습니다. R3는 검수, 보정, 재확인, 품질 게이트에 가까울 수 있습니다. 이름이 무엇이든 중요한 것은 둘을 같은 것으로 가정하지 않는 태도입니다.

R1이 성공했다고 R3가 성공한 것은 아닙니다. R1이 Draft를 만들었지만 R3가 그 Draft를 못 읽으면 검수는 실패입니다. R3가 검수안을 냈지만 R1이나 보정 단계가 그 내용을 반영하지 못하면 결과는 바뀌지 않습니다. 자동화는 한 단계의 성공이 아니라 단계 사이 연결로 판단해야 합니다.

그런데 사람은 같은 화면을 보면 같은 맥락을 공유한다고 생각하기 쉽습니다. 디스코드 채널에 리서치가 보이고, 검수 메시지가 보이고, 작성 요청이 보이면 모두가 같은 자료를 보고 있다고 느낍니다. 하지만 에이전트는 실제 입력으로 받은 것만 봅니다. 보이는 것과 전달된 것은 다릅니다.

그래서 최종 자동 재인증에서는 R1과 R3가 각각 무엇을 읽고, 무엇을 만들고, 무엇을 다음 단계로 넘겼는지 확인해야 합니다. 인증은 시작점이고, 핸드오프는 본게임입니다.

## 자동 재인증이 필요한 순간

### 세션 만료는 늘 예고하고 오지 않는다

세션 만료는 친절하지 않습니다. 어떤 날은 몇 시간 동안 문제없이 작동하고, 어떤 날은 갑자기 끊깁니다. 토큰이 만료됐거나, 쿠키가 갱신됐거나, 외부 서비스 정책이 바뀌었거나, 브라우저 세션이 분리됐을 수 있습니다. 사용자는 똑같이 요청했는데 결과가 달라집니다.

자동 재인증이 필요한 이유는 여기 있습니다. 매번 사람이 로그인 상태를 확인하고, 권한을 다시 보고, 실패하면 수동으로 고치는 것은 오래가기 어렵습니다. 하네스가 최소한의 상태 확인을 해주고, 필요한 경우 재인증 흐름으로 보내야 합니다.

하지만 자동 재인증은 조심해야 합니다. 무조건 다시 로그인하거나, 무조건 토큰을 갱신하는 방식은 위험할 수 있습니다. 잘못하면 기존 세션을 깨거나, 다른 계정으로 전환되거나, 불필요한 인증 시도가 반복될 수 있습니다. 그래서 자동 재인증에는 조건이 있어야 합니다.

예를 들어 보호된 작업을 시도했는데 권한 오류가 나는지, R1과 R3 중 어느 경로에서만 실패하는지, 실패가 일시적인 네트워크 문제인지 실제 인증 문제인지 구분해야 합니다. 자동화는 빠르게 움직일수록 멈춤 조건도 분명해야 합니다.

### 권한 오류와 맥락 오류를 구분해야 한다

자동화 실패를 보면 모두 인증 문제처럼 보일 때가 있습니다. 하지만 실제로는 권한 오류가 아니라 맥락 오류일 수 있습니다. 파일을 못 읽는 것이 권한 때문일 수도 있지만, 애초에 Draft 경로가 잘못 전달됐을 수도 있습니다. 검수가 실패한 것이 인증 때문이 아니라 본문이 아직 없어서일 수도 있습니다.

이번 하네스 사고가 딱 그런 교훈을 줬습니다. Claude7 검수 시점이 본문 작성 전이었습니다. 그러니 검수는 형식상 있었지만 실제 본문 품질을 확인한 것이 아니었습니다. 이건 인증 문제가 아니라 순서 문제였습니다. 아직 Draft가 없는데 검수를 시킨 것이 핵심 원인이었습니다.

그래서 자동 재인증 하네스는 실패를 곧바로 “인증 만료”로 단정하면 안 됩니다. 먼저 질문해야 합니다. 입력이 있었나. Draft가 있었나. 검수자가 실제 본문을 읽었나. 권한이 막힌 것인가, 대상이 없었던 것인가. 이 구분이 없으면 재인증을 여러 번 해도 같은 문제가 반복됩니다.

사실 좋은 하네스는 재인증도 잘하지만, 재인증할 필요가 없는 문제를 재인증으로 오해하지 않는 쪽이 더 중요합니다. 순서 문제는 순서로 고쳐야 하고, 권한 문제는 권한으로 고쳐야 합니다.

## 이번 하네스 사고가 알려준 가장 큰 교훈

### 본문 전 검수는 검수가 아니다

이번 작업 흐름에서 가장 분명했던 문제는 Claude7 검수 시점이었습니다. 검수 단계가 있었습니다. 검수 모델도 있었습니다. 그런데 검수 대상인 본문이 아직 없었습니다. 그러니 검수는 실제 품질 확인이 아니라 이름만 있는 단계였습니다.

이 문제는 자동화에서 정말 자주 생깁니다. 단계 이름만 보면 모든 것이 갖춰져 보입니다. 리서치, 작성, 검수, 보정, 발행 판단이 순서대로 있는 것처럼 보입니다. 그런데 실제 데이터 흐름이 맞지 않으면 아무 의미가 없습니다. 검수는 결과물이 생긴 뒤에 해야 합니다.

솔직히 사람이 손으로 일할 때는 이 순서를 잘 압니다. 글을 쓰고, 저장하고, 읽고, 고칩니다. 그런데 여러 모델과 세션이 붙으면 이 당연한 순서가 쉽게 흐려집니다. 검수 모델이 먼저 실행되고, 작성 모델이 뒤늦게 본문을 만들고, 품질 게이트가 엉뚱한 통계를 기준으로 돌아갈 수 있습니다.

그래서 이번에는 Draft 저장 후 검수로 바꿨습니다. 먼저 저장 가능한 본문을 만들고, 그 본문을 기준으로 Claude7이 검수합니다. 검수 결과를 다시 반영하고, 마지막으로 품질 게이트를 확인합니다. 작은 변경처럼 보이지만, 하네스 품질에는 큰 차이를 만듭니다.

### 재인증도 대상이 있어야 의미가 있다

검수에 대상이 필요하듯 재인증에도 대상이 필요합니다. “로그인 됐나?”만 보는 것은 약합니다. 실제 보호된 작업을 할 수 있는지 봐야 합니다. R1이 Draft를 만들 수 있는지, R3가 그 Draft를 읽을 수 있는지, 보정 단계가 수정본을 만들 수 있는지 확인해야 합니다.

즉, 재인증 테스트는 추상 상태를 보는 것이 아니라 실제 작업을 통과시키는 과정이어야 합니다. R1에서 초안 생성이 되는가. R3에서 검수가 되는가. 품질 게이트가 돌아가는가. 외부 발행은 요청 없을 때 멈추는가. 이 질문을 통과해야 “재인증이 의미 있다”고 말할 수 있습니다.

그런데 여기서도 과하게 자동화하면 위험합니다. 인증이 안 됐다고 바로 외부 로그인 창을 열거나, 메시지를 보내거나, 계정을 바꾸는 것은 조심해야 합니다. 내부 점검과 외부 행동은 구분해야 합니다. 자동 재인증은 내부 상태를 회복하는 장치이지, 사용자 승인 없이 외부 행동을 밀어붙이는 장치가 아닙니다.

이번 글도 같은 원칙을 따릅니다. 네이버나 인스타 자동 발행은 하지 않습니다. 파일도 직접 쓰지 않습니다. 저장 가능한 Markdown 초안만 출력합니다. 하네스가 할 수 있는 일과 지금 해야 하는 일을 구분하는 것이 중요합니다.

## 최종 자동 재인증의 기본 흐름

### 먼저 상태를 읽고, 그다음 재인증한다

자동 재인증은 무조건 다시 인증하는 것으로 시작하면 안 됩니다. 먼저 현재 상태를 읽어야 합니다. 어떤 경로가 실패했는지, 어떤 권한이 필요한지, 마지막 성공 시점이 언제였는지, 실패 로그가 무엇인지 확인해야 합니다.

예를 들어 R1은 정상인데 R3만 실패한다면 전체 재인증보다 R3 경로의 권한과 입력을 보는 것이 먼저입니다. 반대로 R1과 R3 모두 실패한다면 공통 인증이나 외부 서비스 상태를 의심할 수 있습니다. 이 차이를 보지 않고 무조건 재인증하면 문제를 가릴 수 있습니다.

사실 재인증은 치료보다 진단이 먼저입니다. 어디가 끊겼는지 모르면 다시 연결해도 또 끊깁니다. 그래서 하네스는 실패를 세분화해야 합니다. 인증 실패, 권한 부족, 입력 누락, Draft 없음, 검수 대상 없음, 품질 게이트 실패를 따로 기록해야 합니다.

이렇게 상태를 읽은 뒤에 필요한 재인증만 실행해야 합니다. 작은 문제에는 작은 조치가 맞습니다. 모든 실패를 큰 재인증으로 처리하면 자동화가 무거워지고, 진짜 원인도 흐려집니다.

### 재인증 뒤에는 실제 작업으로 확인한다

재인증을 했으면 마지막에는 실제 작업으로 확인해야 합니다. “성공” 메시지만 믿으면 안 됩니다. R1이 간단한 Draft를 만들 수 있는지, R3가 그 Draft를 검수할 수 있는지, 결과가 다음 단계로 넘어가는지 확인해야 합니다.

이 확인은 길 필요가 없습니다. 짧은 테스트 글, 작은 파일, 간단한 읽기 요청, 구조 검수 정도면 충분할 수 있습니다. 중요한 것은 실제 흐름을 통과한다는 점입니다. 인증만 살아 있고 작업이 안 되면 재인증은 끝난 것이 아닙니다.

자동화에서는 완료 메시지가 너무 쉽게 나옵니다. “재인증 완료”, “검수 완료”, “작성 완료” 같은 말은 편하지만, 산출물 없이는 약합니다. 그래서 완료 뒤에는 항상 증거가 있어야 합니다. 파일 경로, Draft 본문, 검수 결과, 품질 게이트 통과 여부 같은 것들입니다.

R1 R3 최종 자동 재인증의 핵심도 이것입니다. 완료 보고보다 산출물 확인이 먼저입니다. 인증이 됐다고 말하기보다, 인증된 상태로 무엇을 할 수 있는지 보여줘야 합니다.

## R1 경로에서 확인해야 할 것

### 초안 생성과 입력 반영

R1 경로에서는 초안 생성 능력을 확인해야 합니다. 사용자가 준 주제, 요구사항, 날짜, 태그, 분량, 구조 조건이 제대로 반영되는지 봐야 합니다. 특히 블로그 하네스에서는 YAML frontmatter와 H2/H3 구조가 기본입니다.

R1이 정상이라면 저장 가능한 Markdown 본문을 만들 수 있어야 합니다. 오늘 날짜가 맞아야 하고, 제목이 주제와 맞아야 하고, H2마다 H3가 있어야 합니다. 전체 H3 개수는 H2 이상이어야 합니다. 코드펜스 금지 같은 출력 조건도 지켜야 합니다.

그런데 R1이 초안을 만든다고 해서 모든 것이 끝난 것은 아닙니다. 초안은 초안입니다. 검수 전입니다. R1이 만든 결과물은 R3가 읽을 수 있는 형태로 넘어가야 합니다. 즉, R1의 성공 조건에는 “다음 단계가 이어받을 수 있는 산출물”이 포함됩니다.

이 부분이 자동화의 실전 감각입니다. 사람에게 읽기 좋은 답변과 하네스가 이어받기 좋은 산출물은 다를 수 있습니다. R1은 둘 다 만족해야 합니다.

### 파일 직접 쓰기 금지 조건 확인

이번 요청처럼 “파일을 직접 쓰지 말고 출력만 하라”는 조건은 반드시 지켜야 합니다. 자동화 도구는 파일을 쓸 수 있어도, 사용자가 금지하면 쓰면 안 됩니다. 할 수 있는 것과 해도 되는 것은 다릅니다.

R1 재인증 테스트에서도 이 조건을 확인해야 합니다. R1이 파일 쓰기 권한을 갖고 있는지보다, 파일 쓰기 금지 요청을 지키는지가 더 중요할 때가 있습니다. 권한이 있다고 해서 항상 실행하는 하네스는 위험합니다.

특히 블로그 작업에서는 이런 조건이 자주 나옵니다. 초안만 출력하라, 코드펜스를 넣지 말라, 네이버나 인스타에 자동 발행하지 말라, GitHub Pages용 Markdown만 만들라 같은 조건입니다. R1은 이런 조건을 정확히 반영해야 합니다.

사실 자동화의 신뢰는 이런 사소한 조건에서 쌓입니다. 큰 기능을 잘해도 작은 금지 조건을 어기면 사용자는 불안해집니다. R1은 첫 단계에서 신뢰를 잃지 않아야 합니다.

## R3 경로에서 확인해야 할 것

### 실제 Draft를 읽었는지 확인하기

R3 경로의 핵심은 검수입니다. 그런데 검수는 반드시 실제 Draft를 읽은 뒤에 해야 합니다. 파일을 못 읽었거나, 본문이 아직 없거나, 메타데이터만 본 상태라면 최종 검수로 취급하면 안 됩니다.

이번 하네스 사고에서 Claude7 검수가 무의미했던 이유도 여기에 있습니다. 검수 모델이 있었지만 본문 전이었습니다. 실제 Draft가 없었기 때문에 문장 흐름, 요구사항 누락, 구조 문제를 제대로 볼 수 없었습니다. 이걸 검수 완료로 보면 위험합니다.

R3가 해야 할 일은 명확합니다. 검수 대상이 무엇인지 밝힙니다. 그 대상이 실제 Draft인지 확인합니다. 구조, 길이, 요구사항, 금지 조건을 봅니다. 그리고 수정 권고를 다음 단계가 반영할 수 있는 형태로 남깁니다.

검수 결과는 추상적이면 안 됩니다. “좋습니다”보다 “H2 8개, H3 10개, 길이 기준 통과, 발행 금지 조건 유지”가 더 쓸모 있습니다. 자동화에서는 구체적인 검수 로그가 곧 신뢰입니다.

### 보정 단계로 이어지는지 확인하기

R3가 검수안을 냈다고 끝이 아닙니다. 그 검수안이 보정 단계로 이어져야 합니다. “고쳐야 합니다”라는 말과 “고친 전체 본문이 나왔습니다”는 다릅니다. 자동화에서는 이 차이가 자주 사고를 만듭니다.

예를 들어 R3가 “길이 초과”를 지적했는데 보정 단계가 실제로 줄이지 않으면 아무 의미가 없습니다. R3가 “본문 전 검수는 무의미”라고 지적했는데 다음 실행에서도 Draft 전 검수를 반복하면 교훈이 반영되지 않은 것입니다.

그래서 R3 재인증 테스트는 검수 결과의 전달까지 봐야 합니다. R3가 만든 권고가 R1이나 보정 에이전트에게 전달되는지, 수정본에 실제로 반영되는지, 최종 품질 게이트에서 통과하는지 확인해야 합니다.

솔직히 검수만 많고 반영이 없는 하네스는 사람을 더 피곤하게 합니다. 지적은 쌓이는데 결과물은 안 바뀌기 때문입니다. 좋은 R3는 지적하는 경로가 아니라 결과물을 개선하는 경로여야 합니다.

## 자동 재인증에서 멈춤 조건이 중요한 이유

### 외부 발행은 가장 늦게 움직여야 한다

자동 재인증이 성공했다고 바로 외부 발행까지 가면 위험합니다. 인증은 내부 상태이고, 발행은 외부 행동입니다. 둘은 분리해야 합니다. 네이버, 인스타, GitHub Pages, 디스코드 공지, 이메일 같은 외부 행동은 사용자의 명확한 요청과 조건 확인 뒤에만 해야 합니다.

이번 요청은 초안 작성입니다. 네이버나 인스타 자동 발행은 하지 말라고 되어 있습니다. 파일도 직접 쓰지 말라고 되어 있습니다. 그러니 이 단계에서는 Markdown 본문만 출력해야 합니다. 재인증 테스트가 성공했다고 해도 발행으로 넘어가면 안 됩니다.

이 원칙은 답답해 보일 수 있지만 안전합니다. 자동화가 강해질수록 멈춤 조건은 더 중요해집니다. 할 수 있는 일의 범위가 넓어질수록, 하지 말아야 할 일을 더 분명히 해야 합니다.

사실 좋은 하네스는 빠르게 진행하는 장치이면서 동시에 잘 멈추는 장치입니다. 특히 외부로 나가는 작업은 마지막까지 보류하는 편이 안전합니다. 내부 검수와 외부 발행은 다른 단계입니다.

### 실패하면 다시 인증하기보다 원인을 남긴다

재인증 테스트가 실패했을 때 바로 다시 시도하는 것은 쉬운 선택입니다. 하지만 실패 원인을 남기지 않으면 같은 문제가 반복됩니다. 자동화에서 반복 실패는 대부분 기록 부족에서 커집니다.

실패 로그에는 최소한 네 가지가 있어야 합니다. 어떤 경로인지, 어떤 작업인지, 어떤 오류인지, 다음 조치가 무엇인지입니다. 예를 들어 “R3 Draft 읽기 실패: Draft 경로 없음, R1 산출물 전달 확인 필요”처럼 적으면 다음 행동이 보입니다.

반대로 “재인증 실패”라고만 적으면 아무것도 알 수 없습니다. 인증이 문제인지, Draft가 문제인지, 입력이 문제인지, 권한이 문제인지 알 수 없습니다. 그러면 사람은 다시 처음부터 확인해야 합니다.

재인증은 성공보다 실패 로그가 더 중요할 때도 있습니다. 실패가 잘 보이면 고칠 수 있습니다. 실패가 흐리면 자동화는 계속 불안합니다.

## 최종 자동 재인증을 운영 규칙으로 만들기

### 최소 실행 루프를 정한다

R1 R3 최종 자동 재인증은 최소 실행 루프를 가져야 합니다. 루프가 있어야 매번 같은 방식으로 확인할 수 있습니다. 임기응변으로 하면 오늘은 되고 내일은 안 됩니다.

간단한 루프는 이렇습니다. 입력 번들을 확인합니다. R1이 Draft를 생성합니다. Draft 존재를 확인합니다. R3가 Draft를 기준으로 검수합니다. 보정 단계가 검수안을 반영합니다. 품질 게이트가 구조와 길이를 확인합니다. 외부 발행 조건을 확인하고, 요청이 없으면 멈춥니다.

이 흐름은 복잡해 보이지만 실제로는 기본입니다. 작성 전 검수 금지, Draft 없는 보정 금지, 검수 없는 발행 금지, 요청 없는 외부 행동 금지. 이 네 줄만 지켜도 자동화 사고가 크게 줄어듭니다.

그런데 이 루프는 문서로 남겨야 합니다. 사람이 머릿속으로만 기억하면 언젠가 빠집니다. 하네스 규칙, 작업 로그, 체크리스트에 남겨야 다음에도 같은 방식으로 실행할 수 있습니다.

### 재인증 결과를 짧게 요약한다

재인증 결과는 길게 쓰지 않아도 됩니다. 하지만 핵심은 빠지면 안 됩니다. R1 상태, R3 상태, Draft 상태, 검수 상태, 품질 게이트 상태, 발행 상태를 짧게 남기면 충분합니다.

예를 들어 이렇게 남길 수 있습니다. R1 초안 생성 가능. R3 Draft 기준 검수 가능. 품질 게이트 길이 통과. 네이버와 인스타 발행 보류. 파일 직접 쓰기 없음. 이런 요약은 나중에 문제를 찾을 때 도움이 됩니다.

특히 “발행 보류”를 명시하는 것이 중요합니다. 자동화에서는 아무것도 하지 않은 것도 중요한 결정일 수 있습니다. 요청이 초안이면 발행하지 않는 것이 성공입니다. 사용자가 발행을 원하지 않았는데 발행하지 않은 것은 정상 동작입니다.

이런 로그가 쌓이면 하네스에 대한 신뢰가 생깁니다. 그냥 “잘 됨”이 아니라 무엇이 잘 됐는지 보이기 때문입니다.

## 결론: R1 R3 재인증은 인증보다 흐름 검증이다

### 최종 확인은 산출물로 해야 한다

R1 R3 최종 자동 재인증의 결론은 분명합니다. 인증이 됐다는 말보다, 인증된 상태로 실제 산출물이 만들어지고 검수되고 보정되는지가 중요합니다. 자동화에서 말은 쉽고, 산출물은 어렵습니다. 그래서 산출물로 확인해야 합니다.

R1은 저장 가능한 Draft를 만들 수 있어야 합니다. R3는 그 Draft를 실제로 읽고 검수할 수 있어야 합니다. 보정 단계는 검수안을 반영해야 합니다. 품질 게이트는 길이와 구조를 확인해야 합니다. 외부 발행은 요청이 없으면 멈춰야 합니다. 이 전체가 이어져야 최종 재인증이 의미를 갖습니다.

이번 하네스 사고는 좋은 경고였습니다. Claude7 검수 시점이 본문 전이라 무의미했고, Draft 저장 후 검수로 바꿨습니다. 그 교훈은 재인증에도 그대로 적용됩니다. 대상 없는 검수는 검수가 아니고, 실제 작업 없는 인증은 약합니다.

솔직히 자동화는 점점 더 많은 일을 하게 됩니다. 그래서 더 중요한 것은 속도가 아니라 순서입니다. 순서가 맞으면 도구가 강해지고, 순서가 틀리면 좋은 모델도 헛돌게 됩니다.

### 이번 글은 초안으로만 남긴다

이 글은 R1 R3 최종 자동 재인증을 설명하는 저장 가능한 Markdown 초안입니다. 파일을 직접 쓰지 않고, 네이버나 인스타에 자동 발행하지 않습니다. 지금 단계의 목표는 발행이 아니라 Draft로 남길 수 있는 본문을 만드는 것입니다.

이 조건을 지키는 것도 하네스 테스트의 일부입니다. 사용자가 하지 말라고 한 일을 하지 않는 것. 외부 발행을 보류하는 것. 코드펜스를 넣지 않는 것. 오늘 날짜를 frontmatter에 넣는 것. 이런 작은 조건들이 모여 전체 신뢰를 만듭니다.

결국 R1 R3 최종 자동 재인증은 “다시 로그인했다”가 아니라 “다시 일할 수 있는 상태가 됐다”를 확인하는 과정입니다. 그리고 일할 수 있는 상태란, 입력을 받고, Draft를 만들고, Draft 뒤에 검수하고, 보정하고, 품질 게이트를 통과하고, 외부 행동은 조건을 확인한 뒤에만 하는 상태입니다. 이 순서가 살아 있으면 하네스는 믿을 수 있습니다.
