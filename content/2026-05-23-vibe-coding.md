## 바이브코딩이 갑자기 뜬 이유

### "코드를 쓰는 일"보다 "원하는 걸 말하는 일"이 앞에 왔다

요즘 X에서 개발자들 사이에 자주 보이는 말이 있습니다. **바이브코딩**, 영어로는 vibe coding입니다. 말 그대로 분위기 타듯이 코딩한다는 뜻에 가깝습니다. 엄밀한 설계 문서부터 쓰고, 함수 하나하나 직접 치고, 테스트를 직접 정리하는 방식과는 조금 다릅니다.

바이브코딩은 이런 흐름입니다.

"이런 앱 만들어줘."
"버튼을 오른쪽으로 옮겨줘."
"에러가 났어. 고쳐줘."
"디자인 좀 더 SaaS처럼 바꿔줘."
"이제 로그인 붙여줘."

사람은 요구사항과 피드백을 말하고, AI가 코드를 만듭니다. 개발자는 매번 코드를 직접 쓰기보다 결과를 보고 다시 지시합니다. 그래서 처음 보면 진짜 신기합니다. 예전에는 작은 웹앱 하나 만들려면 HTML, CSS, JavaScript, 백엔드, 배포까지 알아야 했는데... 이제는 말로 꽤 많은 걸 만들 수 있습니다.

그런데 여기서 중요한 질문이 생깁니다. 이게 정말 개발일까요? 아니면 그냥 AI에게 맡기는 실험일까요? 이 질문 때문에 바이브코딩은 지금 재미있는 유행이면서 동시에 꽤 위험한 논쟁거리입니다.

### 시작점은 Karpathy의 X 글이었다

바이브코딩이라는 말을 크게 퍼뜨린 건 Andrej Karpathy의 X 글입니다. 2025년 2월 2일, Karpathy는 X에서 "vibe coding"이라는 표현을 쓰며, LLM이 너무 좋아져서 코드가 존재한다는 사실까지 잊고 흐름에 맡기는 새로운 코딩 방식이라고 설명했습니다. 그 글에서 그는 Cursor Composer와 Sonnet, SuperWhisper를 언급했고, "I just see stuff, say stuff, run stuff, and copy paste stuff, and it mostly works"라고 썼습니다.

출처는 직접 확인했습니다.

- Andrej Karpathy X post, 2025-02-02: https://x.com/karpathy/status/1886192184808149383
- 확인 시점에 X 페이지에는 약 7.1M views가 표시됐습니다.
- Ars Technica도 2025-03-05 기사에서 이 X 글을 인용하며 바이브코딩을 "AI가 쓴 코드를 이해하지 않고 받아들이는 방식이 인기를 얻고 있다"는 흐름으로 다뤘습니다.
- Wikipedia의 Vibe coding 문서도 이 용어가 2025년 2월 Karpathy의 글에서 나왔다고 정리합니다.

사실 이 지점이 바이브코딩의 핵심입니다. 코드를 세세하게 이해하고 통제하기보다, 결과를 보고 말로 계속 조정하는 방식입니다.

### X에서 보이는 흐름은 "재미"와 "실전"이 섞여 있다

X에서 바이브코딩은 크게 세 가지 방향으로 보입니다.

첫째, 빠른 프로토타입입니다. 아이디어가 떠오르면 Cursor, Claude, Replit 같은 도구로 바로 만들어봅니다. 며칠짜리 사이드 프로젝트를 몇 시간 안에 화면으로 보는 식입니다.

둘째, 비개발자 진입입니다. Replit CEO Amjad Masad는 X에서 "Vibe coding is already here. 75% of Replit customers never write a single line of code!!"라고 썼습니다. 이 글은 Karpathy의 원글을 인용한 것이고, 2025년 2월 3일 게시됐습니다.

- Amjad Masad X post, 2025-02-03: https://x.com/amasad/status/1886516600653930924
- 확인 시점에 X 페이지에는 약 209.1K views가 표시됐습니다.

셋째, 실험 콘텐츠입니다. Peter Yang은 2025년 3월 4일 X에서 Cursor, Sonnet 3.7, SuperWhisper로 블록 스타일 좀비 FPS를 만들어보겠다는 스레드를 올렸습니다. 이건 바이브코딩이 단순한 업무 자동화가 아니라, 만들기 과정을 공개하는 콘텐츠가 되고 있다는 신호입니다.

- Peter Yang X post, 2025-03-04: https://x.com/petergyang/status/1896793172489155048
- 확인 시점에 X 페이지에는 약 779.1K views가 표시됐습니다.

즉, 요즘 바이브코딩 트렌드는 단순히 "AI가 코드 짜준다"가 아닙니다. **아이디어를 빠르게 제품 화면으로 바꾸고, 그 과정을 공유하고, 비개발자도 개발 흐름에 들어오는 문화**에 가깝습니다.

## 바이브코딩은 왜 매력적인가

### 아이디어에서 화면까지 가는 시간이 짧아진다

예전에는 아이디어가 있어도 막히는 지점이 많았습니다. 로그인은 어떻게 붙이지? 데이터베이스는 뭘 쓰지? UI는 어떻게 짜지? 배포는 어디에 하지? 이런 질문이 하나씩 벽처럼 나왔습니다.

바이브코딩은 이 벽을 낮춥니다. 사람이 모든 걸 직접 몰라도 일단 화면을 만들 수 있습니다. 특히 작은 도구, 랜딩 페이지, 내부 대시보드, 프로토타입, 개인 자동화에는 효과가 큽니다. 완벽하지 않아도 일단 만져볼 수 있으니까요.

이게 정말 큽니다. 아이디어는 머릿속에 있을 때와 화면에 있을 때 완전히 다릅니다. 화면이 생기면 피드백을 줄 수 있습니다. "이건 아닌데?", "여기 버튼이 필요하네", "모바일에서 깨지네" 같은 판단이 가능해집니다. 바이브코딩은 이 피드백 사이클을 빠르게 만듭니다.

근데 빠르다는 건 항상 좋기만 할까요? 여기서부터 조심해야 합니다.

### 비개발자도 제품 대화에 들어올 수 있다

바이브코딩이 재미있는 이유는 개발자만의 일이 아니기 때문입니다. 디자이너, 기획자, 마케터, 대표, 크리에이터도 "이런 화면이 필요하다"고 말하면 AI가 기본 구조를 만들어줄 수 있습니다.

예를 들어 카페 사장님이 예약 관리 페이지를 만들고 싶다고 해봅시다. 예전에는 외주를 알아보거나 개발자를 찾아야 했습니다. 지금은 "예약 목록, 고객 이름, 시간, 상태 변경 버튼이 있는 웹페이지 만들어줘"라고 말해서 초안을 만들 수 있습니다. 물론 그대로 운영하면 안 됩니다. 하지만 대화의 출발점으로는 충분합니다...

사실 이게 바이브코딩의 좋은 면입니다. 만들 수 있는 사람이 늘어납니다. 아이디어가 코드 앞에서 멈추지 않습니다.

### 개발자에게도 귀찮은 작업을 줄여준다

바이브코딩은 비개발자만을 위한 게 아닙니다. 개발자에게도 꽤 유용합니다. 특히 반복적이고 귀찮은 UI 수정, 초기 scaffolding, 작은 스크립트, 테스트 데이터 생성, 문서용 예제 코드 같은 작업에서 좋습니다.

Karpathy가 예로 든 "sidebar padding을 절반으로 줄여줘" 같은 요청이 딱 그렇습니다. 직접 찾아서 고칠 수도 있지만, 귀찮습니다. AI에게 말하고 결과를 보면 됩니다. 개발자는 더 큰 구조와 판단에 집중할 수 있습니다.

다만 여기서 중요한 선이 있습니다. 개발자가 코드를 이해하고 리뷰하면서 쓰면 생산성 도구입니다. 그런데 코드를 읽지 않고 "Accept All"만 반복하면 위험이 커집니다. 이 차이를 구분해야 합니다.

## 요즘 바이브코딩 트렌드

### 트렌드 1: 음성으로 만드는 개발

Karpathy의 원글에서 눈에 띄는 부분은 SuperWhisper입니다. 그는 Composer와 대화하듯이 말하면서 코딩하고, 키보드를 거의 만지지 않는다고 했습니다. Peter Yang의 X 글도 Cursor + Sonnet 3.7 + SuperWhisper 조합을 언급합니다.

이 흐름은 중요합니다. 개발이 키보드 중심에서 말하기 중심으로 일부 이동하고 있다는 뜻이니까요. 물론 완전히 그렇게 바뀐다는 말은 아닙니다. 하지만 프로토타입 단계에서는 말로 지시하고 결과를 보는 방식이 점점 자연스러워지고 있습니다.

개발자가 아닌 사람에게는 이게 더 큽니다. 코드를 몰라도 말로 요구사항을 설명할 수 있으니까요. 그런데 말로 만드는 개발은 요구사항이 흐려지기 쉽습니다. "좀 더 예쁘게" 같은 말은 사람마다 다르게 해석됩니다. 그래서 바이브코딩을 잘하려면 오히려 말이 더 정확해야 합니다...

### 트렌드 2: Cursor, Replit, Claude 조합

X와 기사에서 반복적으로 보이는 도구는 Cursor, Replit, Claude 계열입니다. Karpathy는 Cursor Composer와 Sonnet을 언급했고, Amjad Masad는 Replit 고객의 75%가 한 줄도 코드를 쓰지 않는다고 말했습니다. Business Insider도 Replit, Cursor, Sonnet 조합을 중심으로 바이브코딩 흐름을 설명합니다.

이 조합이 뜨는 이유는 단순합니다. 에디터, 실행 환경, AI 모델이 한곳에 붙어 있기 때문입니다. AI가 코드를 만들고, 바로 실행하고, 에러를 다시 넣고, 고치는 흐름이 빠릅니다.

예전에는 AI에게 코드를 받고, 로컬에 붙여넣고, 에러를 확인하고, 다시 복사해서 물어봤습니다. 지금은 그 과정이 훨씬 짧아졌습니다. 이게 바이브코딩을 유행으로 만든 기술적 배경입니다...

### 트렌드 3: "소프트웨어 for one"과 개인 도구

New York Times의 Kevin Roose는 바이브코딩으로 개인용 작은 앱들을 만들어보며 "software for one"이라는 표현을 썼습니다. 모두에게 팔 제품이 아니라, 나 하나를 위한 소프트웨어라는 뜻입니다.

이 관점은 꽤 중요합니다. 바이브코딩은 처음부터 대규모 SaaS를 만드는 방식이라기보다, 나에게 필요한 작은 도구를 빠르게 만드는 데 잘 맞습니다. 예를 들어 개인 일정 정리, 블로그 소재 관리, 이미지 프롬프트 생성기, 가게 운영 체크리스트, 간단한 계산기 같은 것들입니다.

사실 이런 도구는 예전에도 만들 수 있었습니다. 문제는 비용이었습니다. 개발자를 붙이기엔 작고, 직접 만들기엔 귀찮았습니다. 바이브코딩은 이 빈 공간을 채웁니다.

### 트렌드 4: 공개 제작 콘텐츠

Peter Yang의 사례처럼, 바이브코딩은 결과물만이 아니라 만드는 과정 자체가 콘텐츠가 됩니다. X에서 "오늘 밤 바이브코딩으로 이런 게임 만들어볼게"라고 선언하고, 진행 과정을 스레드로 보여주는 방식입니다.

이건 개발 문화에도 영향을 줍니다. 예전에는 완성된 제품이나 코드 저장소를 공유했다면, 이제는 만드는 과정의 시행착오도 콘텐츠가 됩니다. 에러가 나고, AI가 고치고, 다시 망가지고, 또 고치는 과정이 하나의 쇼가 됩니다.

그런데 이 방식은 과장되기 쉽습니다. 2시간 만에 앱을 만들었다는 말은 매력적이지만, 그 앱이 안전한지, 유지보수 가능한지, 실제 사용자에게 배포해도 되는지는 별도 문제입니다.

## 바이브코딩의 위험한 지점

### 코드를 모르면 문제가 생겼을 때 멈춘다

바이브코딩의 가장 큰 위험은 이해 부족입니다. AI가 코드를 만들어줬는데 내가 그 코드를 이해하지 못하면, 문제가 생겼을 때 어디를 봐야 할지 모릅니다.

Ars Technica는 이 흐름을 다루면서, AI가 쓴 코드를 이해하지 않고 받아들이는 방식이 인기를 얻고 있지만 실제 애플리케이션에 안정적으로 쓸 수 있는지에는 질문이 남는다고 지적했습니다. Simon Willison도 "LLM이 모든 코드를 썼더라도, 리뷰하고 테스트하고 이해했다면 그건 내 기준의 vibe coding이 아니라 typing assistant로 쓴 것"이라는 취지로 말했습니다.

이 구분이 중요합니다. AI가 코드를 쓰는 것 자체가 문제는 아닙니다. 문제는 사람이 이해와 검증을 포기하는 순간입니다.

### 보안과 개인정보는 장난이 아니다

바이브코딩으로 만든 앱이 개인용 장난감이면 괜찮을 수 있습니다. 그런데 로그인, 결제, 고객 데이터, 개인정보, 회사 내부 자료가 들어가면 이야기가 달라집니다.

AI가 만든 코드에는 보안 취약점이 있을 수 있습니다. 인증이 약할 수 있고, 데이터 검증이 빠질 수 있고, API 키가 노출될 수 있고, 권한 처리가 틀릴 수 있습니다. 겉으로 화면은 잘 돌아가도 내부가 위험할 수 있습니다.

특히 초보자는 "작동한다"를 "안전하다"로 착각하기 쉽습니다. 작동과 안전은 다릅니다. 화면이 뜨는 것과 운영 가능한 것은 다릅니다. 이걸 놓치면 작은 장난감 프로젝트가 실제 피해로 이어질 수 있습니다.

### 유지보수는 나중에 온다

바이브코딩은 시작이 빠릅니다. 그런데 유지보수는 나중에 옵니다. 처음에는 AI가 잘 만들어준 것처럼 보이지만, 기능이 늘어나고 파일이 많아지고 예외 상황이 생기면 복잡해집니다.

그래서 바이브코딩은 "빠르게 만들기"에는 좋지만 "오래 운영하기"에는 규칙이 필요합니다. 폴더 구조, 테스트, 환경변수, 배포 방식, 보안 점검, 로그 확인 같은 기본기가 있어야 합니다.

## 바이브코딩을 제대로 쓰는 기준

### 프로토타입과 운영 제품을 구분해야 한다

바이브코딩을 무조건 피할 필요는 없습니다. 오히려 잘 쓰면 엄청 유용합니다. 단, **프로토타입**과 **운영 제품**을 구분해야 합니다.

프로토타입은 빠르게 만들어도 됩니다. 아이디어를 확인하고, 화면을 보고, 동작을 실험하는 단계입니다. 이때는 속도가 중요합니다. 완벽한 코드보다 빠른 피드백이 더 중요할 수 있습니다.

운영 제품은 다릅니다. 사용자 데이터가 들어가고, 결제가 붙고, 회사 업무에 연결되고, 장애가 나면 문제가 되는 단계입니다. 이때는 바이브만으로 가면 안 됩니다. 코드 리뷰, 테스트, 보안 점검, 백업, 로그, 배포 절차가 필요합니다.

한 문장으로 정리하면 이렇습니다.

**바이브코딩은 시작을 빠르게 만드는 방식이지, 검증을 생략해도 되는 면허가 아닙니다.**

### Accept All은 습관이 되면 위험하다

Karpathy의 원글에는 "Accept All always, I don't read the diffs anymore"라는 표현이 나옵니다. 이 문장이 바이브코딩의 재미를 잘 보여주지만, 동시에 위험도 보여줍니다.

개인 장난감 프로젝트라면 괜찮을 수 있습니다. 그런데 회사 코드, 고객 데이터, 결제, 자동화 스크립트에 이 습관이 들어가면 위험합니다. AI가 바꾼 코드가 무엇인지 모르고 받아들이면, 나중에 문제가 생겼을 때 책임은 사람에게 남습니다.

그래서 저는 실무에서는 이렇게 나누는 게 맞다고 봅니다.

| 상황 | Accept All 가능성 |
|---|---|
| 버리는 프로토타입 | 가능 |
| 개인 실험 | 제한적으로 가능 |
| 블로그 데모 코드 | 검토 후 가능 |
| 회사 내부 도구 | diff 확인 필요 |
| 고객 데이터 처리 | 리뷰와 테스트 필수 |
| 결제/인증/권한 | 보안 점검 필수 |

귀찮아도 이 선은 필요합니다.

### 좋은 프롬프트보다 좋은 검증 루프가 중요하다

바이브코딩을 잘하려면 프롬프트를 잘 쓰는 것도 중요합니다. 하지만 더 중요한 건 검증 루프입니다.

예를 들어 이렇게 해야 합니다.

1. 작게 요청한다.
2. 바로 실행한다.
3. 에러를 확인한다.
4. 변경 diff를 본다.
5. 테스트를 추가한다.
6. 다시 실행한다.
7. 배포 전 체크리스트를 본다.

이 흐름이 있으면 바이브코딩은 생산성 도구가 됩니다. 반대로 검증 없이 큰 기능을 한 번에 맡기면 도박이 됩니다.

사실 프롬프트는 누구나 늘릴 수 있습니다. "전문가처럼 해줘", "완벽하게 해줘", "버그 없게 해줘" 같은 말은 쉽습니다. 그런데 실제 품질은 그런 말보다 실행과 검증에서 나옵니다.

## X 데이터로 본 현재 분위기

### 긍정 흐름: 누구나 만들 수 있다는 흥분

X에서 가장 강한 긍정 흐름은 "나도 만들 수 있다"입니다. Karpathy의 원글은 7.1M views를 기록했고, Amjad Masad의 Replit 관련 글도 209.1K views가 표시됐습니다. Peter Yang의 바이브코딩 게임 제작 글도 779.1K views가 표시됐습니다.

이 숫자만 봐도 바이브코딩이 개발자 내부의 작은 농담을 넘어, 넓은 AI 개발 문화로 퍼졌다는 걸 알 수 있습니다. 특히 X에서는 짧은 데모, 빠른 결과물, 실시간 제작 스레드가 잘 퍼집니다. 바이브코딩과 잘 맞는 환경입니다.

### 중립 흐름: 프로토타입에는 좋다

중립적인 흐름도 있습니다. 바이브코딩은 버리는 프로젝트, 주말 실험, 개인 도구, 데모에는 좋다는 관점입니다. Karpathy도 원글에서 "throwaway weekend projects"에는 나쁘지 않다고 했습니다.

저도 이 관점이 가장 현실적이라고 봅니다. 처음부터 운영 제품을 만들겠다고 들어가면 위험하지만, 아이디어를 확인하는 데는 좋습니다. 말 그대로 바이브를 타고 빠르게 확인하는 겁니다.

### 부정 흐름: 이해 없는 코드는 위험하다

부정적인 흐름은 유지보수와 보안입니다. Ars Technica는 실제 애플리케이션에 안정적으로 쓸 수 있는지 의문을 제기했고, Business Insider도 기술 부채와 보안 취약점이 생길 수 있다는 전문가 의견을 소개했습니다.

이건 단순한 보수적 반응이 아닙니다. 실제로 AI가 만든 코드를 이해하지 못하면 문제가 생겼을 때 대응이 어렵습니다. 바이브코딩이 쉬워질수록, 검증의 중요성도 같이 커집니다.

## 그래서 바이브코딩을 어떻게 써야 할까

### 개인 프로젝트는 과감하게, 공개 서비스는 조심스럽게

개인 프로젝트라면 바이브코딩을 과감하게 써도 됩니다. 작은 도구, 블로그 보조 스크립트, 대시보드 초안, 아이디어 검증에는 정말 좋습니다. 실패해도 큰 피해가 없고, 배움이 남습니다.

하지만 공개 서비스라면 조심해야 합니다. 특히 로그인, 결제, 개인정보, 파일 업로드, 관리자 권한, 외부 API 키가 들어가면 반드시 검토해야 합니다. 작동한다고 바로 공개하면 안 됩니다.

여기서 기준은 간단합니다.

**내 혼자 쓰는가, 남의 데이터가 들어가는가?**

남의 데이터가 들어가는 순간부터 바이브코딩은 초안 도구가 되고, 최종 책임은 사람이 져야 합니다.

### 비개발자는 작은 문제부터 시작해야 한다

비개발자라면 처음부터 큰 서비스를 만들려고 하지 않는 게 좋습니다. 작은 문제부터 시작하세요.

예를 들어 업무 체크리스트, 간단한 계산기, 블로그 제목 후보 생성기, 이미지 프롬프트 정리 도구 같은 것들입니다. 이런 작은 도구는 바이브코딩과 잘 맞습니다. 만들면서 배우고, 망가지면 다시 만들 수 있습니다.

## 마무리: 바이브는 좋지만 책임은 남는다

### 바이브코딩은 새로운 장난감이 아니라 새 작업 방식이다

바이브코딩은 단순한 유행어가 아닙니다. 아이디어를 제품 화면으로 바꾸는 방식이 바뀌고 있다는 신호입니다. 말로 만들고, 결과를 보고, 다시 지시하고, 빠르게 고치는 흐름은 앞으로 더 자연스러워질 겁니다.

그런데 이 흐름이 강해질수록 책임도 중요해집니다. AI가 만들었다고 책임이 사라지지 않습니다. 사용자에게 보여주는 순간, 데이터를 받는 순간, 배포하는 순간 책임은 사람에게 옵니다.

### 좋은 바이브코딩은 검증을 포함한다

저는 바이브코딩을 무조건 찬성하거나 반대하지 않습니다. 잘 쓰면 정말 좋습니다. 특히 프로토타입, 개인 도구, 학습, 아이디어 검증에는 강력합니다. 그런데 검증 없이 운영으로 밀어 넣으면 위험합니다.

좋은 바이브코딩은 이런 모습이어야 합니다.

- 빠르게 만든다.
- 바로 실행한다.
- diff를 본다.
- 테스트한다.
- 보안을 확인한다.
- 이해 안 되는 부분은 다시 묻는다.
- 운영 전에는 사람이 책임지고 검토한다.

이 정도 기준만 있어도 바이브코딩은 훨씬 건강해집니다.

### 결국 핵심은 "말로 만든 뒤, 눈으로 검증하기"

바이브코딩은 말로 만드는 시대를 보여줍니다. 하지만 말로 만들었다고 끝은 아닙니다. 눈으로 보고, 손으로 확인하고, 테스트로 검증해야 합니다.

처음엔 신기해서 계속 맡기고 싶습니다. 저도 그렇습니다. 그런데 한 번 해보시면 알아요. 바이브코딩의 진짜 실력은 프롬프트를 길게 쓰는 데서 나오지 않습니다. AI가 만든 결과를 얼마나 잘 판단하고, 어디까지 믿고, 어디서 멈출지 정하는 데서 나옵니다.

정리하면 이렇습니다. 바이브코딩은 **개발을 쉽게 시작하게 해주는 방식**입니다. 하지만 좋은 제품을 만들려면 여전히 이해, 검증, 책임이 필요합니다. 바이브는 시작을 빠르게 만들고, 검증은 끝을 안전하게 만듭니다. 이 둘을 같이 가져가야 합니다.
